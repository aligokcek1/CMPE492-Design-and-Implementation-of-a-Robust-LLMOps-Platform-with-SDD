## 1. Lightning AI Python SDK — Deployment, Status, Delete

### Decision
Use the `lightning-sdk` package (`pip install lightning-sdk`) to interact with Lightning AI's managed cloud platform programmatically.

### Findings

The `lightning-sdk` package exposes a `LightningClient` that wraps the Lightning AI REST API. Key calls relevant to this feature:

```python
from lightning_sdk import LightningClient, Machine

# Authenticate — API key read from env var or passed explicitly
client = LightningClient(api_key="<USER_API_KEY>")

# Deploy a LitServe server to Lightning AI cloud
app = client.run_server(
    script="server.py",          # path to LitServe server script
    name="llmops-<model-slug>",  # unique deployment name
    machine=Machine.T4,          # cheapest GPU — NVIDIA T4 (16 GB VRAM)
)
# Returns: App object with .id (deployment ID) and .url (endpoint URL, available after running)

# Poll status
app = client.get_app(app_id=lightning_ai_deployment_id)
# app.status: "running" | "starting" | "failed" | "stopped"
# app.url: str | None

# Delete / stop
client.delete_app(app_id=lightning_ai_deployment_id)

# Validate key — a lightweight API call that fails on bad auth
client.list_apps()  # raises LightningSDKException on invalid key
```

**Status mapping** (Lightning AI → platform):

| Lightning AI status | Platform status | Notes |
|---|---|---|
| `starting` | `deploying` | Equivalent to GKE "deploying" |
| `running` | `running` | Endpoint URL is now populated |
| `failed` | `failed` | SDK populates `app.message` with error |
| `stopped` | `deleted` | After `delete_app` completes |

**Auth errors**: The SDK raises `lightning_sdk.exceptions.LightningSDKException` with an `UNAUTHORIZED` code on invalid API key. This maps to `LightningAIAuthError` in the platform's provider layer, which flips `validation_status = "invalid"` on the credential row (mirrors `GCPAuthError` handling).

### Provider Interface (for test injection)

```python
class LightningAIProvider(Protocol):
    async def deploy(self, *, hf_model_id: str, api_key: str) -> tuple[str, str | None]:
        """Submit deployment. Returns (lightning_ai_deployment_id, endpoint_url_or_None)."""

    async def get_status(self, *, deployment_id: str, api_key: str) -> tuple[str, str]:
        """Poll status. Returns (platform_status, message)."""

    async def delete(self, *, deployment_id: str, api_key: str) -> None:
        """Stop/delete a deployment. No-op if already gone (treat as success)."""

    async def validate_api_key(self, *, api_key: str) -> None:
        """Raise LightningAIAuthError if key is invalid."""
```

### Rationale
- `lightning-sdk` is the official Python client maintained by Lightning AI; avoids subprocess fragility and subprocess-based fake providers in tests.
- The provider interface follows the identical pattern as `GCPProvider` — enables a `FakeLightningAIProvider` for contract tests that never call the real API.

### Risk / Validation
- Exact method names and exception class names must be verified against the installed `lightning-sdk` version during the Red phase of each TDD cycle.
- The machine type constant (`Machine.T4`) targets the NVIDIA T4 (16 GB VRAM) — the cheapest GPU tier on Lightning AI. Verify the constant name against the installed SDK version; fall back to a string `"T4"` if needed. Models must fit within 16 GB VRAM at fp16/bf16.

### Alternatives Considered
- **CLI subprocess** (`lightning deploy server.py --cloud`): brittle, cannot be easily faked for tests, stdout parsing is fragile. Rejected.
- **Direct REST API via httpx**: would require re-implementing auth token refresh and endpoint discovery. Rejected in favour of the official SDK.

## 2. LitServe + vLLM Server Script for Arbitrary HF Models

### Decision
Generate a Python server script string at deploy time — parameterised by `hf_model_id` — that uses `litserve` with `vllm` as the inference backend and exposes an OpenAI-compatible `/v1/chat/completions` endpoint.

### Findings

LitServe supports vLLM natively via `ls.LitAPI` or via the bundled OpenAI spec compatibility layer. The generated script follows this pattern:

```python
# Auto-generated by llmops-platform — DO NOT EDIT
import os
import litserve as ls
from vllm import LLM, SamplingParams

HF_MODEL_ID = os.environ.get("HF_MODEL_ID", "{hf_model_id}")

class VLLMInferenceAPI(ls.LitAPI):
    def setup(self, device):
        self.llm = LLM(model=HF_MODEL_ID)

    def decode_request(self, request):
        messages = request.get("messages", [])
        prompt = "\n".join(m["content"] for m in messages)
        return prompt

    def predict(self, prompt):
        params = SamplingParams(temperature=0.7, max_tokens=512)
        outputs = self.llm.generate([prompt], params)
        return outputs[0].outputs[0].text

    def encode_response(self, output):
        return {
            "choices": [
                {
                    "message": {"role": "assistant", "content": output},
                    "finish_reason": "stop",
                }
            ]
        }

if __name__ == "__main__":
    server = ls.LitServer(VLLMInferenceAPI(), accelerator="gpu", max_batch_size=1)
    server.run(port=8000)
```

The script is passed to `client.run_server(script="<temp_file_path>")`. The platform writes the script to a temporary file, submits it, then deletes the temp file.

**Module**: `backend/src/services/litserve_gpu.py` — exposes `generate(hf_model_id: str) -> str` (returns script string). The script is written to a `NamedTemporaryFile` by the orchestrator before the SDK call.

**Why a string generator (not a file on disk)**: The server script is model-specific and ephemeral — it only needs to exist long enough for the SDK upload call. Keeping it in memory avoids leftover files and is consistent with how `vllm_manifest.py` works (returns a YAML string, not a file).

**OpenAI compatibility**: The generated endpoint is OpenAI-compatible at `/v1/chat/completions` — the same format expected by the existing inference proxy (`inference_proxy.py`).

### Rationale
- LitServe + vLLM is the stack specified in the feature (FR-006).
- Generating the script programmatically (not storing a static template file) keeps the deployment path fully parameterised without requiring a template engine.

### Alternatives Considered
- **LitServe OpenAI spec built-in**: LitServe has `ls.OpenAISpec()` which may provide a cleaner interface; evaluate during implementation and use if it reduces boilerplate.
- **vLLM's own OpenAI-compatible server** (without LitServe): contradicts the spec (FR-006 requires LitServe). Rejected.

## 3. SQLite Structural Migration for Nullable Columns

### Decision
Rebuild the `deployments` table within a transaction to make `gcp_project_id`, `gke_cluster_name`, and `gke_region` nullable, and to add `hardware_type` and `lightning_ai_deployment_id` columns.

### Findings

SQLite does not support `ALTER TABLE … ALTER COLUMN` to change nullability. The standard approach is:

```sql
BEGIN TRANSACTION;
CREATE TABLE deployments_new (
    id TEXT PRIMARY KEY,
    user_id TEXT NOT NULL,
    hf_model_id TEXT NOT NULL,
    hf_model_display_name TEXT NOT NULL,
    hardware_type TEXT NOT NULL DEFAULT 'cpu',
    gcp_project_id TEXT UNIQUE,          -- was NOT NULL
    gke_cluster_name TEXT,               -- was NOT NULL
    gke_region TEXT,                     -- was NOT NULL
    lightning_ai_deployment_id TEXT,     -- new, nullable
    status TEXT NOT NULL DEFAULT 'queued',
    status_message TEXT,
    endpoint_url TEXT,
    created_at DATETIME NOT NULL,
    updated_at DATETIME NOT NULL,
    deleted_at DATETIME
);
INSERT INTO deployments_new SELECT
    id, user_id, hf_model_id, hf_model_display_name,
    'cpu',                               -- hardware_type default
    gcp_project_id, gke_cluster_name, gke_region,
    NULL,                                -- lightning_ai_deployment_id
    status, status_message, endpoint_url,
    created_at, updated_at, deleted_at
FROM deployments;
DROP TABLE deployments;
ALTER TABLE deployments_new RENAME TO deployments;
CREATE INDEX ix_deployments_user_id ON deployments (user_id);
CREATE INDEX ix_deployments_user_status ON deployments (user_id, status);
COMMIT;
```

This runs only when the existing `deployments` table is missing the `hardware_type` column (checked via `inspector.get_columns`). Safe for both fresh installs (handled by `Base.metadata.create_all`) and upgrades.

### Risk
- **Data safety**: The migration runs inside a transaction; a failure rolls back completely, leaving the old table intact.
- **Fresh installs**: `create_all` creates the new table definition directly — no rebuild needed.
- **Testing**: Contract test `conftest.py` creates a fresh DB every test run via `ensure_schema()`, so no upgrade path is exercised in tests (intentional — kept simple per constitution VI).

## 5. Frontend Hardware Selector Widget

### Decision
Use Streamlit `st.radio()` with `horizontal=True` for the CPU / GPU selector in `render_public_repo_deploy_section`. The radio widget returns `None` when no option is selected only if `index=None` is passed (Streamlit ≥ 1.28). The Deploy button's `disabled` parameter is set to `hardware_type is None or info is None`.

```python
hardware_type = st.radio(
    "Hardware",
    options=["CPU", "GPU"],
    index=None,          # no default selection
    horizontal=True,
    key="public_repo_hardware_selector",
)
deploy_disabled = hardware_type is None or info is None
st.button("🚀 Deploy", disabled=deploy_disabled, ...)
```

### Rationale
`st.radio(index=None)` is the canonical Streamlit pattern for "required choice" — the button stays disabled until the user actively selects. This satisfies FR-001, FR-002, and FR-009 with zero custom state management.
